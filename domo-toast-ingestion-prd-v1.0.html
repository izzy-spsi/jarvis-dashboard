PRD Domo Ops v1.0

Domo Toast Data Ingestion + Shared DB ACP

Owner: Hermes · Stakeholders: JarvisDomo, OpenClaw, Staff OS v3.1, Future Agents · Status: Draft

1 · Problem

Multiple apps — OpenClaw, Hermes, Staff OS — need live Toast data. Right now they either: read from a shared cache, call Toast independently, or show local seed data. That creates rate-limit contention, stale reads, schema drift, and credential sprawl. We need one canonical ingestion layer and a shared Postgres schema designed for analytics, not just CRUD. We also need an Agent Control Protocol so future Hermes/OpenClaw agents can read/write safely without stepping on each other.

Hard constraint: Zero fake/demo data in production. If live Toast is unavailable, the UI must show the ingestion status + last successful sync time, not invented numbers.

2 · Goals

  • G1 Single writer: Hermes becomes the sole Toast ingestion agent. All other apps read from shared Postgres.
  • G2 Analytics-first schema: Store Toast Standard + Analytics payloads in a schema optimized for reporting, reconciliation, and downstream apps.
  • G3 ACP: Agents access data through scoped API routes + row-level intent, not direct DB credentials.
  • G4 Multi-source ready: Schema accommodates Toast, future POS, labor, and guest data sources as first-class citizens.
  • G5 Reconciliation: Daily sales snapshots can be reconciled against Orders API, Analytics API, and daily email totals.

3 · Non-Goals

  • Replacing the existing Domo Ops UI/app — this is a backend/data layer PRD only.
  • Building real-time streaming — 15-minute sync cadence is sufficient for operations.
  • Building a new auth system — reuse existing Domo Ops NextAuth/JWT layer.
  • Merging OpenClaw and Hermes codebases — ACP lets them coexist without merging.

5 · Smart Shared DB Schema

Design principle: store raw external payloads as immutable snapshots, then derive operator-facing views from them. This preserves auditability and lets multiple agents read safely.

5.1 Core models

ModelPurposeKey fieldsSource
ExternalSource Registry of trusted data providers id, name, type, authRef, configJson, isActive, lastSyncAt Admin config
ExternalDataJob Tracks every ingestion run id, sourceId, startedAt, finishedAt, status, records, error, payloadRef Hermes runner
ExternalDataRow Raw normalized row from any source id, jobId, sourceId, businessDate, category, entityType, payload, createdAt Ingestion pipeline
ToastDailyMetric Derived daily sales row from Analytics + Standard reconciliation id, businessDate, restaurantGuid, netSales, grossSales, tax, tips, discounts, voids, refunds, checks, covers, avgOrderValue, revenueCenter, diningOption, orderSource, hourlyLaborHours, hourlyLaborCost, salesPerLaborHour, rawAnalyticsJson, rawStandardJson, reconciledAt, syncJobId Hermes backfill
ToastHourlyMetric Same as daily, but hourly grain id, businessDate, businessHour, restaurantGuid, ... Hermes backfill
ToastMenuItem Menu master from Analytics + Standard id, externalId, name, category, price, revenueCenter, lastSyncedAt Hermes backfill
ToastEmployee Employee roster from Standard + labor analytics id, toastGuid, firstName, lastName, email, phone, jobTitle, hourlyRate, lastSyncedAt Hermes backfill
ToastPayout Payout/deposit reporting id, payoutId, settledDate, salesDate, amount, fees, net, method, restaurantGuid, rawJson Hermes backfill
ToastGuestPayment Guest/payment reporting id, paymentId, businessDate, lastFour, amount, type, restaurantGuid, rawJson Hermes backfill
AgentAccessGrant ACP: which agents can read which tables/rows id, agentId, sourceId, tables, filtersJson, issuedAt, expiresAt, revokedAt, issuedBy Admin / Hermes cert

5.2 Schema rules

  • Raw payloads are immutable. Never update or delete ExternalDataRow.payload, ToastDailyMetric.rawAnalyticsJson, or ToastDailyMetric.rawStandardJson after insert.
  • Derived tables are idempotent. Re-running the same Hermes sync for the same date/restaurant must upsert, not duplicate.
  • Source isolation. Every derived row points to syncJobId so we can trace it back to a run.
  • Future-proofing. Adding a new source (e.g., QuickBooks, Fishbowl) means adding a new ExternalSource.type and optional derived models — no schema migration required.

6 · Agent Control Protocol (ACP)

Future agents need data access without receiving raw DB credentials. ACP defines a token-scoped, route-level contract between agents and the shared DB. Hermes issues short-lived grants; the API enforces them.

6.1 Participants

  • ACP Issuer: Hermes agent. Owns creation/revocation of grants.
  • ACP Gateway: Shared app API layer (`/api/acp/**`). Enforces grants.
  • ACP Consumer: Any agent needing data: OpenClaw, JarvisDomo, future analytics/marketing agents.

6.2 Grant model

FieldMeaning
agentIdOpaque caller identity, e.g. openclaw:main, hermes:domo
tablesAllowed models, e.g. ["ToastDailyMetric","ToastHourlyMetric"]
filtersJsonRow-level filter, e.g. {"restaurantGuid":""}
expiresAtTTL. Default 24h for interactive, 1h for batch.

6.3 ACP routes

RouteMethodAuthBehavior
/api/acp/queryPOSTACP token + HMAC sigRead-only SELECT against allowed tables with enforced filter
/api/acp/ingestPOSTACP token + Hermes certWrite normalized rows to ExternalDataRow + derived tables
/api/acp/statusGETACP tokenReturn last sync per source + record counts
/api/acp/grantsPOST/GET/DELETEIssuer key onlyCreate/list/revoke grants
Principle: If an agent cannot present a valid ACP grant, it gets 401 with no details. It should not learn table names.

7 · Hermes Ingestion Runner Design

Hermes owns a single, scheduled runner that:

  1. Authenticates to Toast Standard + Analytics APIs using separate credentials stored outside the app.
  2. Requests Analytics for day time range per restaurant GUID.
  3. Polls /era/v1/metrics/{reportRequestGuid} with backoff.
  4. Normalizes payload into ExternalDataRow, then upserts ToastDailyMetric.
  5. Emits ToastSyncLog with run metadata.
  6. Exposes /api/acp/status so other apps know whether data is live or stale.

7.1 Rate-limit strategy

  • Use /era/v1/metrics/day for daily grand opening backfill.
  • Respect Analytics limits: 10 req/min for day/week.
  • Reuse Standard API reads for menu/employees/payouts — separate quota.
  • Cache restaurant-info for 24h.

10 · Verification

  • /api/acp/status returns last sync time + record counts per source.
  • Hermes runner logs one ExternalDataJob per run with records > 0.
  • ToastDailyMetric daily totals match Toast daily email within tolerance.
  • Consumer agents can query through ACP but fail with 401 if grant revoked.
  • Re-running the same backfill date does not duplicate rows (@@unique([businessDate, restaurantGuid, revenueCenter, diningOption, orderSource])).
